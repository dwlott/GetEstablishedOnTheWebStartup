# Workspace

## Purpose

`Workspace/` holds **your** project-specific material: goals, preferences, and
future working files. It stays separate from reusable instructions in
`Capabilities/` and `Standards/`.

## Core Boot Files (Starter)

| File | Purpose |
| --- | --- |
| `OwnerGoals.md` | Practical outcomes from Quick Startup |
| `OwnerPreferences.md` | Connector, intake, Git adoption, and import settings |
| `ValuableReferences.md` | Candidate external links and sources |

Additional child folders appear when you have real content or run Capability
**Setup** prompts. See
[Capabilities/RepositoryScaffold/IntendedRepositoryTree.md](../Capabilities/RepositoryScaffold/IntendedRepositoryTree.md).

## Owner Intake (when you have files)

During Quick Startup or later sessions, agents may save **your** source material
here:

| Folder | Use for |
| --- | --- |
| `Documents/` | Resume, business plan drafts, prior notes, PDFs, `.docx` imports |
| `Assets/` | Logos, photos, brand files (when you add them) |

These folders are **not** in the shipped starter ZIP — create them when needed.
Record paths in `OwnerPreferences.md` Import section.

Do **not** put owner uploads in `Workspace/Reference/` or ad hoc folder names.

## Agent Notes

- Treat Workspace as protected owner content.
- Do not invent owner facts, offers, proof, or platform decisions.
- Do not overwrite owner files without explicit approval — except
  [Workspace Adoption Prep](../Capabilities/GettingStarted/WorkspaceAdoptionPrep.md)
  stale scaffold reset at Quick Startup.
